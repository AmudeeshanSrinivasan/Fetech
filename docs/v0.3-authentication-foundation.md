# v0.3 authentication and structured API conformance

Fetech v0.3 closes all 23 capabilities across authentication/private sessions and structured
APIs/feeds. The conformance overlay reports 21 native paths, two optional configured connectors, no
planned gaps, and `closure_ready=true`.

| Category | Native | Optional | Total |
| --- | ---: | ---: | ---: |
| Authentication and private sessions | 8 | 2 | 10 |
| Structured APIs and feeds | 13 | 0 | 13 |
| **v0.3** | **21** | **2** | **23** |

The optional paths are `sso` and `private_workspace`. Their typed session/connector contracts,
policy enforcement, outcomes, and failure semantics ship in core, while an operator must explicitly
configure the identity or workspace connector.

## Credential and session contracts

Callers inject a provider and put only an opaque lookup reference in `FetchRequest`:

```python
from fetech import (
    CredentialMaterial,
    FetechClient,
    FetchRequest,
    InMemoryCredentialProvider,
    InMemorySessionProvider,
    OriginScopedSession,
)

reference = "vault://service/account"
provider = InMemoryCredentialProvider(
    {
        reference: CredentialMaterial.bearer(
            "https://api.example.com",
            token_from_secret_store,
        )
    }
)
sessions = InMemorySessionProvider(
    {
        reference: OriginScopedSession.oauth(
            "https://api.example.com",
            reference,
            issuer_origin="https://identity.example.com",
            scopes=("read:data",),
        )
    }
)

async with FetechClient(
    credential_provider=provider,
    session_provider=sessions,
) as client:
    result = await client.fetch(
        FetchRequest(
            target="https://api.example.com/data",
            output_requirements=("oauth", "rest"),
            authentication_ref=reference,
        )
    )
```

`InMemoryCredentialProvider` and `InMemorySessionProvider` are intended for explicit library use and
tests. Daemon integrations implement the same asynchronous protocols against configured secret and
session stores. A refreshable credential provider may also implement `refresh(reference)`; Fetech
invokes it at most once for GET or HEAD only when a matching OAuth/SSO descriptor authorizes bearer
refresh.

`OriginScopedSession` provides network-free, immutable validation contracts for `login_session`,
`oauth`, `sso`, and `private_workspace`. It keeps authentication and refresh references repr-hidden,
requires an exact HTTPS origin, bounds scopes and expiry metadata, and validates the underlying
`CredentialMaterial` type without copying its values. `SessionProvider` is a distinct runtime
boundary: a generic bearer token cannot claim SSO or private-workspace semantics by itself.

Enforcement:

- Destination policy runs before credential resolution or application.
- Scheme, IDNA host, and effective port must match exactly.
- Opaque references and resolved header/cookie material are byte-bounded before use.
- A first-hop mismatch blocks before transport; later cross-origin hops receive no credentials.
- HTTPX client defaults and its cookie jar never retain caller material.
- Crawl `robots.txt` is always unauthenticated.
- Public requests never consult the provider.
- Unknown credential references return `AUTH_REQUIRED`; missing or unavailable high-level session
  providers return `DEPENDENCY_MISSING`.
- Known expiry uses `auth_expired`. Refresh emits only sanitized lifecycle events.
- Authenticated HTTP/3 remains unavailable because the subprocess has no secret-safe channel.
- Private workspaces require the explicit `private` privacy profile and a connector identity.

## Approved forms and CSRF

Form fields are supplied through an injected `FormSubmissionProvider`, not through request metadata
or plan parameters. The provider atomically consumes a bounded immutable proposal for one run,
keyed by the same opaque authentication reference.

```python
from fetech import (
    FormSubmission,
    FormSubmissionApproval,
    InMemoryFormSubmissionProvider,
)

action = "https://example.com/session"
proposal = FormSubmission(
    target=action,
    method="POST",
    fields={"username": username, "password": password},
    authentication_ref="vault://service/account",
    approval=FormSubmissionApproval.grant(action, "POST"),
)
forms = InMemoryFormSubmissionProvider(
    {"vault://service/account": proposal}
)
```

The request must separately include `approved_capabilities={"form_submit"}`. The executor checks this
before adapter execution; the auth adapter then rechecks the short-lived grant immediately before
I/O. Both checks are required. GET form proposals are rejected; retrieval parameters belong in the
target URL and still pass normal URL sanitization.

`csrf_token` parses only bounded already-acquired HTML. Hidden-token candidates must belong to one
same-origin form and one exact action/method; missing or ambiguous candidates fail closed. The
derived token stays in the execution context's repr-hidden sensitive state and is attached only to
the approved in-memory body. If a proposal already contains CSRF material, it is accepted only when
the value, action, method, field name, source URL, and source-resource lineage exactly match the
token extracted during the current run.

The packaged CLI and zero-argument daemon/MCP entry points use null providers and therefore fail
closed for authenticated execution. v0.3 operators inject providers through the Python SDK,
`create_app(...)`, or `build_server(...)`; automatic provider discovery is intentionally not part of
the alpha contract.

Request-body redirect rules:

- 303 always becomes GET and drops the body.
- 301/302 after POST becomes GET and drops the body.
- Every body-preserving redirect, including same-origin 307/308, requires a new exact-target
  approval and therefore fails closed in the current run.
- An anonymous approved login POST may carry at most 16 bounded Secure cookies across same-origin
  301/302/303 GET redirects for that request chain only.
- Cookie handoff honors exact origin and path, never applies to `robots.txt`, is destroyed on an
  origin change, and is scrubbed from the returned response.
- Mutating form nodes have zero automatic retries.

Fetech does not automate passwords, MFA, CAPTCHAs, IdP interaction, paywall access, or session
creation outside an explicitly supplied, one-shot approved proposal.

## Structured APIs and feeds

The API adapter has no transport. It consumes a bounded raw artifact with successful HTTP acquisition
provenance, preserving the original `Resource` as authority and the raw artifact as the parent of the
normalized result.

Supported paths:

- `rest`, `graphql`, `json_endpoint`, and `xml_endpoint`;
- `rss`, `atom`, `sitemap_xml`, and `openapi_discovery`;
- `github_api`, `semantic_scholar_api`, `arxiv_api`, `openreview_api`, and
  `crossref_openalex_api`.

The deterministic planner selects an explicitly requested canonical ID. Otherwise `.json` and
`.xml` suffixes select generic handlers, and recognized official named-API routes select their
connector. Only one primary structured capability is scheduled; observed underlying formats receive
separate `CapabilityOutcome` records.

Parser safety:

- body size, node count, nesting, feed records, OpenAPI paths, and JSON list breadth are bounded;
- JSON duplicate keys and non-finite numbers are rejected;
- XML DTD and entity declarations are rejected;
- RSS, Atom, sitemap, GraphQL, and OpenAPI require recognizable roots/envelopes;
- OpenAPI YAML uses `safe_load` and rejects aliases and anchors;
- external OpenAPI references and discovered links are never followed automatically;
- named connectors require an exact approved official origin/path and recognizable response schema;
  the explicit public arXiv HTTP endpoint remains subject to the normal public-HTTP policy;
- normalized artifacts use canonical JSON, typed locators, parser versions, parent lineage, and
  original-source authority.

## Persistence and cache isolation

`authentication_ref` is hidden from model representations and redacted before request persistence.
Resolved credentials, refresh material, approved form fields, and submitted bodies never appear in
plans, attempts, diagnostics, outcomes, events, results, runtime Graphify, or Obsidian.

Authenticated plans have separate public and in-memory transport views. Every query value is
redacted in serialized plans, results, ledger records, runtime graphs, and generated API/crawl/URL
candidate documents, even when the parameter name is unknown. Only the private execution request
retains the raw normalized query, and a deserialized plan must be rebound to the exact matching
request before execution.

`CacheKey.for_request()` returns `public` for public requests and a domain-separated
`auth:sha256:<digest>` partition for authenticated requests. Different references cannot share an
entry, and authenticated entries cannot collide with public entries.

## Failure semantics

| Condition | Result or diagnostic |
| --- | --- |
| Missing/unknown credential reference | `AUTH_REQUIRED` |
| Expired or explicitly rejected material | `AUTH_REQUIRED`, `auth_expired` |
| Credential/session/form provider unavailable | `DEPENDENCY_MISSING` |
| Origin, privacy, approval, or body-redirect violation | `BLOCKED_BY_POLICY` |
| Malformed or over-budget structured payload | `FAILED` or `PARTIAL`, `malformed_api_payload` |
| Named API wrong host or wrong envelope | `FAILED`; no generic passthrough |
| Optional SSO/workspace connector absent | `DEPENDENCY_MISSING` |
