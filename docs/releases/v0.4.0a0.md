# Fetech v0.4.0a0 — draft

> **Unreleased:** this document describes the current v0.4 development overlay. The package
> metadata remains `0.3.0a0`; no v0.4 tag, distribution, checksum set, or GitHub Release exists.

Fetech v0.4.0a0 is planned to close the documents/files, media, and
cache/snapshot/archive categories. The development overlay registers implementation paths for the
final 36 catalogue entries, bringing the cumulative catalogue to 155/155 capabilities across all
13 categories.

## Highlights

- Bounded handling for PDF, scanned-PDF/OCR fallback, DOCX, PPTX, XLSX, CSV, TXT, Markdown, JSON,
  XML, ZIP archives, GitHub raw content, Git LFS pointers, and signature-routed dataset files.
- Preferred offline PDF parsing through exact
  `docling-slim[convert-core,format-pdf,models-local]==2.113.0` when an operator supplies a reviewed
  local artifact bundle; `convert-core` supplies the local converter. The v0.4 reference is
  `docling-project/docling-layout-heron@8f39ad3c0b4c58e9c2d2c84a38465abf757272d8`
  with canonical bundle SHA-256
  `e9aab284777b02541f427ff10ff7e2f1b5656eda04afa3082b9b448d8201bd76`.
  Conversion status, errors, timeout state, page coverage, locators, output, parser components, and
  artifact identity remain subject to Python validation.
- Page-located scanned-PDF OCR and exact-origin Git LFS provider contracts whose results are
  independently type-, size-, locator-, origin-, and digest-validated by Python.
- Bounded image validation, OCR, audio/video metadata, transcripts, YouTube metadata, podcast
  feeds, thumbnails, and redacted EXIF normalization.
- Optional built-in yt-dlp metadata acquisition with canonical YouTube targets, sanitized process
  input/environment, strict HTTPS/public-DNS/redirect/identity-response controls, cumulative network
  budgets, and URL-redacted output.
- Built-in exact-host Wayback snapshot acquisition with bounded availability and capture requests,
  public-address DNS validation, and original-publisher authority.
- Immutable content-addressed snapshots with authentication, policy, language, region, parser, and
  `Vary` partitioning; integrity-checked lookup, conditional refresh, and stale handling.
- Policy-gated cache and archive alternatives that can recover from origin failure without treating
  cache presence as authorization or replacing original publisher authority.
- ZIP/TAR extraction through a bounded worker with traversal, link, nesting, member, expansion, and
  compression-ratio controls.
- Shared wire-byte, decompressed-byte, deadline, and archive-member accounting across acquisition
  and derived artifacts.
- Shared global/per-host admission and pacing for primary HTTP and built-in Wayback requests, plus
  process-level admission for the bounded local yt-dlp worker in development mode.
- A POSIX bootstrap deadline that applies irreversible child resource limits without parent-side
  `preexec_fn`, then gives the executed worker only the remaining total wall budget.
- Fail-closed Linux worker profiles for document, archive, image, offline browser, FFmpeg,
  FFprobe, and Tesseract execution, with selective read-only mounts, bounded private tmpfs,
  delegated cgroup-v2 CPU/memory/PID ceilings, explicit namespaces, `no_new_privileges`, and
  libseccomp filtering.
- Explicit `development_unsandboxed` status outside required mode. Required mode refuses a
  missing backend and refuses local yt-dlp until its multi-host traffic has brokered egress.
- An artifact-bound smoke-evidence collector that verifies wheel RECORD hashes, the lockfile, Git
  source state, exact tool/package versions, content-level outputs, endpoint dates, and bounded
  failures.

An implementation path is not the same as guaranteed local availability. Optional paths fail with
typed `DEPENDENCY_MISSING` results when their reviewed package, executable, or configured provider
is absent. Capability interfaces report implementation and base-runtime availability separately.

## Security and provenance

- Python 3.12 remains authoritative for policy, authorization, budgets, adapter admission,
  artifacts, and persistence. Clingo and SWI-Prolog remain optional proposal and explanation
  backends.
- Every destination and redirect remains subject to SSRF, port, address-class, and HTTPS-downgrade
  checks. Third-party snapshot connectors are restricted to public, unauthenticated targets.
- Worker and provider output is untrusted and schema-validated before content-addressed storage.
- Accepted artifacts remain separate from checked-only artifacts, and derived resources retain
  original-source authority and parent lineage.
- Fetched instructions are untrusted content and never become runtime policy.

## Known alpha constraints

- Required mode implements the reference per-worker Linux boundary for the listed built-in offline
  workers, but it is not yet release evidence until the mandatory `containment-linux` job passes on
  the release commit and the installed delegated systemd unit is verified on the target
  distribution. Development mode and macOS remain explicitly unsandboxed.
- Injected OCR/transcription/Git-LFS/browser providers are not made local-process-safe by the
  built-in profiles; required deployments must use reviewed separately isolated connectors or
  leave them disabled.
- Clingo, SWI-Prolog, and optional curl HTTP/3 processes also remain outside the five covered
  profiles and must be isolated separately or disabled in hostile-input deployments.
- Primary HTTP and Wayback use request-level shared scheduling. Local yt-dlp remains available in
  development mode, but required mode rejects it until an allowlisting egress broker can mediate
  its internal multi-host traffic.
- The Docling path, exact Python dependency, reference model revision, canonical manifest, and
  independent expected bundle digest are implemented. Focused source-tree and separately unpacked
  development-wheel contract/content subsets passed on 2026-07-18 and retained sanitized evidence
  for the exact six-file bundle. The wheel check verified its installed package members against `RECORD` and
  bound the result to wheel SHA-256
  `5ed8ba67fe5393c25ac5c195c54e920e058701ebe629d3b1b45881a7cc3dd7cb`. The final clean tagged v0.4
  wheel must rerun that gate in the release environment against a root-owned read-only copy.
  The smaller pypdf parser remains an observable deterministic fallback.
- The reference model card publishes `apache-2.0`, but that captured identifier is not a legal
  approval. Human review of the exact model files, notices, and redistribution terms remains
  required before publishing a model bundle or container image.
- The exact-version catalog covers all 167 third-party identities in the current universal lock,
  and the v0.4 development SPDX SBOM and dependency-license reports regenerate successfully. The
  published v0.3 artifacts remain immutable and are verified against their historical release
  profile. Dependencies recorded under explicit NVIDIA proprietary/EULA and pypdfium2
  mixed-distribution LicenseRefs still require artifact-level notice and redistribution legal
  review before publication; final reports must also be regenerated from the final v0.4 release
  lock.
- Optional live services and executables require smoke evidence tied to the exact installed tool
  version. Wayback evidence must instead record the test date and exercised endpoint/service,
  because the remote service has no package version controlled by Fetech. Presence in an extra or
  lock file is not evidence that a capability executed successfully.
- The daemon and MCP server remain single-tenant, loopback-oriented interfaces without built-in
  multi-user authentication or artifact ACLs.
- Filesystem CAS and SQLite deployments require operator-defined encryption, retention, quota,
  backup, recovery, and deletion controls.

## Release assets

The eventual tagged release must include:

- Python wheel and source distribution built from the tagged commit;
- SHA-256 checksums for both distributions;
- deterministic SPDX 2.3 JSON SBOM and dependency-license report for the exact locked graph;
- security threat model and competitor/dependency evidence snapshots; and
- a capability conformance report that records exact versions for configured optional tools and
  versioned connectors, plus the test date and endpoint/service for unversioned remote services.

The tracked development-overlay SPDX SBOM and dependency-license report are reproducible evidence
for the current `0.3.0a0` lock plus v0.4 overlay. They are deliberately labeled as unreleased and
must be regenerated from the final v0.4 release lock rather than renamed.
